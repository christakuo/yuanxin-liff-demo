# Task 03A-13：Apps Script CRM 串接修改規格

## 一、規格目的

本規格定義 `leads_潛在會員名單` 新增 CRM 欄位後，Apps Script 與顧問業務大廳的安全串接方式。

本階段只建立修改規格，不直接修改、執行或部署線上 Apps Script。

## 二、本次修改範圍

後續程式修改範圍包括：

- `LeadService.gs`
- `ConsultantLeadService.gs`
- `ReferralService.gs`
- 必要的 Apps Script 測試函式
- 顧問業務大廳名單狀態回傳格式
- 前端顯示相容性檢查

原則上不需修改：

- `SheetService.gs`
- `EventService.gs`
- `ConsultantResourceService.gs`
- Google Sheet 既有欄位位置
- 推薦事件紀錄表結構

## 三、正式 CRM 11 階段

正式 CRM 階段統一使用：

```text
01 新名單
02 待首次聯繫
03 已聯繫
04 培育中
05 已預約諮詢
06 方案評估中
07 已簽約（未付款）
08 已成交（付款完成）
09 暫緩
10 無意願
11 停止聯繫／退訂
```

其中：

- `07 已簽約（未付款）`：已完成會員合約，但尚未確認付款完成。
- `08 已成交（付款完成）`：已確認付款完成，才可認定正式成交。
- `09 暫緩`：目前無法繼續，但仍保留後續追蹤可能。
- `10 無意願`：客戶明確表示目前沒有需求。
- `11 停止聯繫／退訂`：客戶要求停止聯絡或取消行銷授權。

## 四、新名單 CRM 預設值

`LeadService.gs` 的 `createLead()` 建立新名單時，除保留既有欄位外，應加入 CRM 預設值。

最低必要預設值：

```text
CRM階段：01 新名單
CRM階段需人工確認：否
```

若正式 Google Sheet 已有相對應欄位，也應寫入：

```text
CRM階段更新時間：建立名單當下時間
CRM階段更新人：system
```

實際欄位名稱必須與 Google Sheet 第一列標題完全一致，不得自行縮寫或改名。

既有欄位仍保留：

```text
名單狀態：新名單
```

第一階段不得刪除或停止寫入舊 `名單狀態`，避免既有程式或報表立即失效。

## 五、既有名單再次互動規則

既有名單再次進站或產生推薦事件時：

- 更新最近互動時間。
- 更新最後更新時間。
- 不得自動重設 `CRM階段`。
- 不得將既有 CRM 階段改回 `01 新名單`。
- 不得清除人工填寫的 CRM 欄位。
- 不得清除 `CRM階段需人工確認`。
- 不得因不同推薦碼自動改派顧問。
- 推薦保護期與歸屬爭議仍依既有規則處理。

單純瀏覽頁面或再次進站，不代表 CRM 階段已改變。

## 六、舊名單狀態與 CRM 階段的過渡原則

過渡期間同時保留：

```text
名單狀態
CRM階段
```

兩者用途不同：

| 欄位 | 過渡期間用途 |
|---|---|
| `名單狀態` | 維持既有程式、報表與相容性 |
| `CRM階段` | 作為新版 CRM 11 階段的正式欄位 |

過渡期間不得直接刪除、改名或搬移 `名單狀態`。

既有名單若已完成人工轉換：

- 顧問業務大廳優先顯示 `CRM階段`。
- 舊 `名單狀態` 暫時保留原值。
- 不要求兩個欄位每次都自動同步。
- 後續確認所有功能完成轉換後，再另案評估是否停止使用舊欄位。

## 七、顧問業務大廳 API 回傳規格

`ConsultantLeadService.gs` 的 `formatLeadRecordForPortal()` 目前只回傳：

```text
leadStatus
```

且資料來源是：

```text
名單狀態
```

後續應增加清楚且相容的回傳方式：

```javascript
crmStage: normalizeText(record['CRM階段']),
leadStatus:
  normalizeText(record['CRM階段']) ||
  normalizeText(record['名單狀態']),
legacyLeadStatus: normalizeText(record['名單狀態'])
```

規則如下：

1. `crmStage` 明確回傳新版 CRM 階段。
2. `leadStatus` 優先使用 `CRM階段`。
3. 若 `CRM階段` 空白，`leadStatus` 暫時退回舊 `名單狀態`。
4. `legacyLeadStatus` 保留舊狀態，供過渡期檢查。
5. 不立即移除既有 `leadStatus`，避免前端顯示中斷。

## 八、前端顯示相容原則

目前 `index.html` 使用：

```text
lead.leadStatus
```

顯示名單狀態。

第一階段可先由 Apps Script 調整 `leadStatus` 的回傳來源，使前端不需立即大幅改寫即可顯示新版 CRM 階段。

前端應達成：

- 有 `CRM階段` 時顯示正式 11 階段。
- `CRM階段` 空白時顯示舊 `名單狀態`。
- 不因新增 API 欄位造成顧問大廳無法載入。
- 我的推薦名單與團隊名單使用一致的狀態規則。
- 不在第一階段開放顧問直接修改 CRM 階段。

CRM 階段編輯、聯繫紀錄、待辦與跟進功能，應在後續前端任務另行實作。

## 九、重複名單輸出邏輯

目前：

- `LeadService.gs` 有 `listConsultantLeads()`
- `ConsultantLeadService.gs` 有 `listConsultantLeadsForPortal()`

兩處均會整理名單資料，但回傳欄位不完全相同。

後續修改時必須：

1. 同時搜尋兩處名單輸出邏輯。
2. 確認兩處是否仍有實際呼叫。
3. 兩處都採相同的 CRM 階段優先規則。
4. 避免一處顯示新版 CRM 階段，另一處仍顯示舊狀態。
5. 暫時不刪除任何可能仍被使用的相容函式。

## 十、同名函式衝突修正規格

目前發現：

- `LeadService.gs` 定義 `applyReferralProtection()`
- `ReferralService.gs` 也定義 `applyReferralProtection()`
- 兩個函式參數與用途不同

Apps Script 的各 `.gs` 檔案共用全域函式空間，因此同名函式可能互相覆蓋。

後續應先搜尋所有呼叫位置，再分別重新命名。

建議命名：

```text
LeadService.gs
evaluateExistingLeadReferralProtection()
```

```text
ReferralService.gs
evaluateReferralCodeProtection()
```

修正要求：

1. 先確認每個函式的實際用途與所有呼叫位置。
2. 修改函式名稱時同步修改對應呼叫。
3. 不得只改函式宣告而遺漏呼叫位置。
4. 不得改變既有 90 天推薦保護期規則。
5. 不得因修正名稱自動變更顧問歸屬。
6. 必須測試同一顧問、不同顧問、保護期內及保護期後情境。

## 十一、SheetService 保留原則

`SheetService.gs` 已採欄位名稱與實際最後欄進行讀寫，目前不需改成固定欄號。

後續不得：

- 將 CRM 欄位改成固定欄號。
- 假設 CRM 欄位永遠位於固定英文字母欄。
- 使用固定長度陣列取代目前動態欄數機制。
- 因新增欄位而縮小既有讀寫範圍。

應繼續使用：

- `getHeaderMap()`
- `getAllRecords()`
- `appendObjectToSheet()`
- `updateObjectInSheet()`
- `sheet.getLastColumn()`

## 十二、修改前安全措施

正式修改線上 Apps Script 前必須：

1. 確認 Google Sheet 已有可回復備份。
2. 備份線上 Apps Script 現行程式碼。
3. 記錄目前 Web App 部署版本。
4. 不直接刪除任何既有函式。
5. 不在同一步同時修改大量無關功能。
6. 先修改同名函式問題。
7. 再加入新名單 CRM 預設值。
8. 再調整顧問大廳回傳欄位。
9. 每一批修改後分別測試。
10. 全部測試通過後才建立新部署版本。

## 十三、必要測試案例

### 測試一：建立新名單

確認：

- 成功建立名單ID
- 推薦碼與顧問歸屬正確
- `名單狀態` 為 `新名單`
- `CRM階段` 為 `01 新名單`
- `CRM階段需人工確認` 為 `否`
- 推薦事件成功新增
- 未覆蓋其他名單或推薦事件

### 測試二：既有名單再次進站

確認：

- 不重複建立新名單
- 最近互動時間正常更新
- CRM 階段不被重設
- 人工填寫資料不被清除
- 推薦事件新增到底部

### 測試三：顧問業務大廳

確認：

- 顧問可正常登入
- 我的推薦名單正常顯示
- 有 CRM 階段的名單顯示新版階段
- CRM 階段空白時可顯示舊名單狀態
- 團隊名單使用相同狀態規則
- 顧問資源中心仍可正常顯示

### 測試四：推薦保護期

確認：

- 同一顧問推薦不產生錯誤
- 保護期內不同推薦碼不自動改派
- 保護期後不同顧問推薦標記人工審核
- 不出現同名函式覆蓋錯誤

### 測試五：既有功能回歸

確認：

- 推薦事件仍寫入最底部
- 原有名單資料未被覆蓋
- 顧問大廳 API 正常回傳
- 前端沒有 JavaScript 錯誤
- 顧問資源中心不受影響

## 十四、失敗回復原則

若測試發生異常：

1. 停止新版本部署。
2. 保留錯誤畫面與執行記錄。
3. 還原 Apps Script 修改前版本。
4. 必要時還原 Google Sheet 備份。
5. 確認既有 Web App 仍可使用。
6. 修正後重新進行小量測試。
7. 未通過完整測試前不得正式切換。

## 十五、本階段不處理事項

本階段暫不處理：

- 顧問直接編輯 CRM 階段
- 聯繫紀錄新增介面
- 今日待辦介面
- 逾期提醒介面
- 自動培育訊息
- 自動轉換成交階段
- 獎金自動結算
- 完整 CRM 後台
- 大量歷史資料自動轉換

以上功能應拆分至後續任務逐步建置。

## 十六、實作順序

建議後續依序進行：

1. 備份 Apps Script。
2. 修正 `applyReferralProtection()` 同名問題。
3. 執行推薦保護期測試。
4. 在 `createLead()` 加入 CRM 預設值。
5. 執行新名單及既有名單測試。
6. 調整顧問大廳 API 回傳 CRM 階段。
7. 執行顧問大廳與團隊名單測試。
8. 執行推薦事件及顧問資源中心回歸測試。
9. 確認全部正常後建立新部署版本。
10. 完成後另建實作與測試紀錄。

## 十七、本階段結論

第一批 CRM 欄位可以安全接入現有 Apps Script。

後續實作必須維持舊 `名單狀態` 的過渡相容性，並優先解決同名函式風險，再加入新名單 CRM 預設值及顧問業務大廳 CRM 階段回傳。

本階段只建立修改規格，不修改 Google Sheet、Apps Script、前端程式或部署版本。
