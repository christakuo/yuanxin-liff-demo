# Task 03A-8：CRM 11 階段下拉選單與舊資料轉換規則

## 一、任務目的

本文件規劃 `leads_潛在會員名單` 從舊版 7 種名單狀態，轉換為正式 11 階段 CRM 的下拉選單與舊資料轉換規則。

本階段目的：

- 設計 CRM 11 階段下拉選單
- 設計舊 7 狀態轉換到新 11 階段的規則
- 設計人工判斷與例外處理原則
- 降低未來正式修改 Google Sheet 時的資料錯置風險

本階段只做規格，不修改 Google Sheet、不修改 Apps Script、不修改前端程式。

## 二、舊版 7 種名單狀態

目前 `leads_潛在會員名單` 的 `名單狀態` 為舊版 7 種：

```text
新名單
已聯繫
已預約諮詢
已簽約
已成交
無效名單
放棄追蹤
```

舊版狀態的主要問題：

- 無法區分新名單與待首次聯繫
- 無法區分已聯繫與培育中
- 無法表示方案評估中
- 已簽約與付款完成定義不清
- 無效名單混在 CRM 階段中
- 放棄追蹤混合了暫緩、無意願、停止聯繫等不同情境

## 三、正式 11 階段下拉選單

正式 CRM 建議 `CRM階段` 下拉選單採用以下 11 種：

```text
01 新名單
02 待首次聯繫
03 已聯繫
04 培育中
05 已預約諮詢
06 方案評估中
07 已簽約（未付款）
08 已成交（付款完成）
09 暫緩
10 無意願
11 停止聯繫／退訂
```

建議原則：

- 新增 `CRM階段` 欄位，不直接覆蓋原 `名單狀態`
- 原 `名單狀態` 保留作為過渡與追溯
- 所有自動化、顧問待辦與成交統計未來逐步改看 `CRM階段`

## 四、舊狀態轉換規則表

| 舊名單狀態 | 新 CRM 階段建議 | 轉換方式 | 備註 |
|---|---|---|---|
| 新名單 | 01 新名單 或 02 待首次聯繫 | 部分自動、部分人工 | 若尚未分派或剛建立，為 01；若已分派顧問且等待聯繫，為 02 |
| 已聯繫 | 03 已聯繫 或 04 培育中 | 部分自動、部分人工 | 若只有首次聯繫，為 03；若需持續內容培育，為 04 |
| 已預約諮詢 | 05 已預約諮詢 或 06 方案評估中 | 部分自動、部分人工 | 若只是已約時間，為 05；若已完成諮詢並在考慮方案，為 06 |
| 已簽約 | 07 已簽約（未付款） 或 08 已成交（付款完成） | 可依付款欄位輔助判斷 | 若未付款，為 07；若已付款，為 08 |
| 已成交 | 08 已成交（付款完成） | 需檢查付款欄位 | 若付款欄位不完整，不可直接認定付款完成 |
| 無效名單 | 不直接轉 CRM 階段 | 建議移到 `名單有效性` | 應區分測試資料、重複資料、資料錯誤或無法聯繫 |
| 放棄追蹤 | 09 暫緩、10 無意願 或 11 停止聯繫／退訂 | 需人工判斷 | 不可全部自動轉為停止聯繫 |

## 五、自動轉換條件

以下欄位可用來輔助自動判斷 CRM 階段。

### 1. 簽約與付款

```text
是否已簽會員合約 = 是
是否已付款 = 否
```

建議轉為：

```text
07 已簽約（未付款）
```

```text
是否已付款 = 是
```

建議轉為：

```text
08 已成交（付款完成）
```

補充規則：

- 若 `orders_訂單主檔` 有付款狀態與入帳確認日，應優先以訂單資料為準
- 若付款狀態不明，不可直接轉為已成交
- 已簽約但未付款，仍需保留顧問跟進或收款提醒

### 2. 諮詢與方案評估

```text
是否已預約諮詢 = 是
是否已簽會員合約 = 否
```

可初步判斷為：

```text
05 已預約諮詢
```

若另有備註顯示已完成方案說明、價格討論、家人討論或正在考慮，則建議人工轉為：

```text
06 方案評估中
```

### 3. 聯繫紀錄

```text
最近聯繫日期 有值
```

至少可判斷為：

```text
03 已聯繫
```

若同時有：

```text
下次追蹤日期 有值
尚未成交
```

則可能為：

```text
04 培育中
```

或：

```text
06 方案評估中
```

需依備註或顧問判斷確認。

### 4. 停止聯繫

若未來新增停止聯繫相關欄位，且出現：

```text
是否停止聯繫 = 是
```

應優先轉為：

```text
11 停止聯繫／退訂
```

並且停止自動化行銷與顧問主動推廣。

## 六、需要人工確認的情境

以下情境不建議完全自動轉換。

### 1. 新名單是否已進入待首次聯繫

舊狀態 `新名單` 需判斷：

- 是否已分派顧問
- 是否已通知顧問
- 是否已超過首次聯繫期限
- 是否只是剛建立的資料

可能轉為：

```text
01 新名單
02 待首次聯繫
```

### 2. 已聯繫是否進入培育中

舊狀態 `已聯繫` 需判斷：

- 客戶是否有興趣
- 是否需要持續教育內容
- 是否已約下一次跟進
- 是否尚未準備諮詢

可能轉為：

```text
03 已聯繫
04 培育中
```

### 3. 已預約諮詢是否進入方案評估中

舊狀態 `已預約諮詢` 需判斷：

- 是否已完成諮詢
- 是否已說明會員方案
- 是否正在考慮價格、醫院選擇、家人討論或時間安排

可能轉為：

```text
05 已預約諮詢
06 方案評估中
```

### 4. 放棄追蹤的實際原因

舊狀態 `放棄追蹤` 可能代表：

- 客戶只是暫緩
- 客戶明確無意願
- 客戶要求停止聯繫
- 顧問自行放棄
- 客戶長時間未回覆

可能轉為：

```text
09 暫緩
10 無意願
11 停止聯繫／退訂
```

其中 `11 停止聯繫／退訂` 必須有客戶明確要求停止聯絡或取消行銷授權，不可由顧問自行判斷。

### 5. 無效名單的資料品質問題

舊狀態 `無效名單` 需判斷原因：

- 測試資料
- 重複資料
- 電話錯誤
- 假資料
- 無法辨識身份
- 無法聯繫

建議不要轉成 CRM 階段，而是放入：

```text
名單有效性
```

## 七、建議新增輔助欄位

為了安全轉換舊資料，建議新增以下欄位。

### 1. 狀態轉換欄位

- 原名單狀態
- CRM階段
- CRM階段轉換方式
- CRM階段轉換時間
- CRM階段轉換人
- CRM階段需人工確認
- CRM階段人工確認備註

### 2. 資料品質欄位

- 名單有效性
- 無效原因
- 是否測試資料
- 是否重複資料
- 合併至名單ID

### 3. 聯繫保護欄位

- 停止聯繫標記
- 停止聯繫原因
- 停止聯繫時間
- 是否允許LINE訊息
- 是否允許電話聯繫
- 是否允許行銷訊息

## 八、分階段導入建議

### 第一階段：新增 CRM階段，不刪除原名單狀態

做法：

- 保留 `名單狀態`
- 新增 `原名單狀態`
- 新增 `CRM階段`
- 新增轉換方式與人工確認欄位

目的：

避免直接覆蓋舊資料造成回復困難。

### 第二階段：人工檢查舊資料轉換結果

先由系統或人工初步轉換，再人工檢查：

- 新名單
- 已聯繫
- 已預約諮詢
- 已簽約
- 已成交
- 無效名單
- 放棄追蹤

尤其應優先檢查：

- 已簽約
- 已成交
- 放棄追蹤
- 無效名單

### 第三階段：Apps Script 改為寫入 CRM階段

當欄位穩定後，Apps Script 應逐步改為：

- 新名單建立時寫入 `01 新名單`
- 已通知顧問後進入 `02 待首次聯繫`
- 顧問首次聯繫後進入 `03 已聯繫`
- 客戶進入長期培育時進入 `04 培育中`
- 付款完成後進入 `08 已成交（付款完成）`
- 停止聯繫時進入 `11 停止聯繫／退訂`

### 第四階段：前端顧問 CRM 顯示 CRM階段

顧問業務大廳或未來 CRM 頁面應顯示：

- CRM階段
- 最近聯繫日期
- 下次追蹤日期
- 是否逾期
- 停止聯繫標記
- 顧問跟進備註

### 第五階段：確認穩定後，再評估停用原名單狀態

等資料轉換、Apps Script 與前端都穩定後，再評估：

- 是否隱藏舊 `名單狀態`
- 是否改由 `CRM階段` 作為正式狀態
- 是否保留 `原名單狀態` 作為歷史紀錄

## 九、風險提醒

### 1. 不可直接覆蓋舊名單狀態

原因：

- 舊資料可能需要回溯
- Apps Script 可能仍讀取原欄位
- 前端可能仍顯示原欄位

### 2. 不可把已簽約直接當成交

原因：

- 已簽約不一定已付款
- 正式成交定義是付款完成
- 獎金不應只因簽約成立

### 3. 不可把放棄追蹤全部轉成停止聯繫

原因：

- 有些客戶只是暫緩
- 有些客戶只是目前無意願
- 停止聯繫／退訂涉及客戶授權與合規保護

### 4. 不可把無效名單混入 CRM 階段

原因：

- 無效名單是資料品質問題
- CRM 階段應描述銷售與跟進流程
- 兩者混用會影響統計

### 5. 停止聯繫／退訂必須優先保護

一旦客戶要求停止聯繫，系統應優先：

- 停止自動化行銷
- 停止顧問推廣提醒
- 顯示禁止聯絡標記
- 保留停止聯繫時間與原因

## 十、結論與下一步

Task 03A-8 確認：

- 11 階段 CRM 應新增為正式 `CRM階段`
- 舊 `名單狀態` 不應直接覆蓋
- 舊 7 狀態可部分自動轉換，但多數仍需人工確認
- `已成交` 必須以付款完成為正式認定點
- `放棄追蹤` 必須拆分為暫緩、無意願、停止聯繫／退訂
- `無效名單` 應轉為資料品質欄位，而不是 CRM 階段
- 停止聯繫／退訂需建立聯繫保護機制
