# Task 03A-9：Google Sheet 欄位調整清單與實作前檢查

## 一、任務目的

本文件整理 `leads_潛在會員名單` 從現有名單表升級為 CRM 工作表前，需要新增、保留、暫不移除的欄位清單。

本階段目的：

- 整理 `leads_潛在會員名單` 需要新增的欄位
- 明確列出第一階段應保留與暫不移除的欄位
- 明確列出可能影響 Apps Script 的欄位與邏輯
- 明確列出可能影響前端顧問 CRM 的顯示與查詢
- 作為未來正式修改 Google Sheet 前的施工檢查清單

本階段只做規格，不修改 Google Sheet、不修改 Apps Script、不修改前端程式。

## 二、本階段不做的事

本階段不做以下事項：

- 不直接改 Google Sheet
- 不直接設定下拉選單
- 不直接搬移舊資料
- 不修改 Apps Script
- 不修改 `index.html`
- 不修改 `referral.js`
- 不修改 `api/`
- 不部署

## 五、暫不移除欄位

第一階段明確不移除：

- `名單狀態`
- `是否已簽會員合約`
- `是否已付款`
- 既有推薦歸屬欄位
- 既有來源欄位
- 既有成交欄位
- 既有獎金歸屬欄位
- 既有顧問跟進備註
- 既有內部備註

原因：

- Apps Script 可能仍讀取 `名單狀態`
- 前端顧問大廳可能仍顯示舊狀態
- `是否已簽會員合約` 與 `是否已付款` 是判斷 07、08 階段的重要依據
- 推薦歸屬與來源欄位涉及顧問獎金與保護期
- 移除欄位會提高正式系統錯誤風險

## 六、Apps Script 可能受影響範圍

未來正式修改 Google Sheet 前，需檢查 Apps Script 是否有使用下列邏輯或欄位。

### 1. 建立名單邏輯

需檢查：

- 建立名單時寫入哪些欄位
- 是否預設寫入 `名單狀態`
- 是否需新增寫入 `CRM階段`
- 是否需寫入 `原名單狀態`
- 是否需寫入 `首次聯繫期限`

### 2. 更新名單邏輯

需檢查：

- 既有名單更新時是否覆蓋狀態
- 推薦碼歸屬是否會改變
- 最近互動時間是否更新
- 是否需更新 `CRM階段更新時間`
- 是否需保留人工修改過的 CRM 階段

### 3. 推薦事件寫入邏輯

需檢查：

- referral_events 是否與 leads 欄位同步
- 是否建立名單
- 是否更新既有名單
- 是否有效推薦碼
- 事件是否應影響 CRM 階段

### 4. 顧問大廳查詢 API

需檢查：

- 顧問名單查詢目前讀取哪個狀態欄位
- 是否讀取 `leadStatus`
- 是否需改為讀取 `CRM階段`
- 是否需同時回傳舊 `名單狀態` 與新 `CRM階段`

### 5. 顧問名單篩選邏輯

需檢查：

- 是否依名單狀態篩選
- 是否依顧問ID篩選
- 是否依團隊權限篩選
- 是否需新增今日待辦、逾期、已簽約未付款等篩選

### 6. 已付款或成交判斷邏輯

需檢查：

- `是否已付款`
- `付款狀態`
- `入帳確認日`
- `orders_訂單主檔`
- `是否列入顧問獎金`
- `獎金歸屬顧問ID`

重點：

```text
已簽約（未付款）不等於已成交（付款完成）
```

### 7. 停止聯繫與退訂保護邏輯

需檢查：

- 是否已有退訂欄位
- 是否已有停止聯繫欄位
- 自動訊息是否會檢查停止聯繫
- 顧問前端是否會顯示禁止聯絡標記

## 七、前端可能受影響範圍

未來正式修改前端前，需檢查以下範圍。

### 1. 顧問業務大廳名單狀態顯示

目前前端可能顯示：

- 名單狀態
- leadStatus
- 顧問推薦名單
- 團隊會員名單

未來需評估改為顯示：

- CRM階段
- 原名單狀態
- 是否需人工確認

### 2. 顧問名單卡片或列表欄位

未來可能需要顯示：

- CRM階段
- 最近聯繫日期
- 下次追蹤日期
- 是否逾期
- 顧問跟進重點
- 顧問跟進備註

### 3. 今日待辦顯示

未來可新增：

- 今日待首次聯繫
- 今日待跟進
- 已逾期未處理
- 即將諮詢
- 已簽約未付款
- 需要人工接手

### 4. 逾期提醒顯示

需依以下欄位判斷：

- 首次聯繫期限
- 是否已首次聯繫
- 下次追蹤日期
- 是否逾期

### 5. 已簽約未付款提示

需顯示：

```text
07 已簽約（未付款）
```

提醒顧問或前台進行付款追蹤，不可列為正式成交。

### 6. 已成交付款完成提示

需顯示：

```text
08 已成交（付款完成）
```

並停止一般成交型推廣訊息，進入會員服務流程。

### 7. 停止聯繫標記

需清楚顯示：

```text
11 停止聯繫／退訂
```

並提醒：

- 不可主動行銷聯繫
- 不可推播促銷訊息
- 需保留停止聯繫紀錄

### 8. 不應受影響的範圍

以下前端功能第一階段不應受影響：

- 會員首頁
- 顧問入口
- 顧問資源中心
- 顧問資源連結
- LINE metadata
- LIFF ID
- LIFF Endpoint URL

## 九、建議實作順序

### 第一步：備份 Google Sheet

正式修改前，先建立完整備份。

備份應包含：

- 所有分頁
- 所有欄位
- 所有下拉選單
- 所有既有資料

### 第二步：新增欄位，不刪除舊欄位

先只新增欄位，不移除既有欄位。

避免影響：

- Apps Script
- 前端顧問大廳
- 推薦事件紀錄
- 既有資料查詢

### 第三步：新增 CRM階段 下拉選單

新增正式 11 階段：

```text
01 新名單
02 待首次聯繫
03 已聯繫
04 培育中
05 已預約諮詢
06 方案評估中
07 已簽約（未付款）
08 已成交（付款完成）
09 暫緩
10 無意願
11 停止聯繫／退訂
```

### 第四步：複製原名單狀態到原名單狀態欄位

將既有 `名單狀態` 複製到：

```text
原名單狀態
```

作為轉換前備份。

### 第五步：小量資料測試轉換

先選少量測試資料進行轉換，例如：

- 新名單
- 已聯繫
- 已預約諮詢
- 已簽約
- 已成交
- 無效名單
- 放棄追蹤

確認轉換邏輯正確後再擴大。

### 第六步：人工檢查高風險狀態

優先人工檢查：

- 已簽約
- 已成交
- 放棄追蹤
- 無效名單

原因：

- 已簽約與已成交涉及付款與獎金
- 放棄追蹤涉及客戶意願與停止聯繫
- 無效名單涉及資料品質

### 第七步：確認 Apps Script 是否需改

檢查 Apps Script 是否仍能：

- 建立名單
- 更新名單
- 寫入推薦事件
- 顯示顧問大廳
- 查詢顧問名單
- 判斷付款完成
- 保護停止聯繫名單

### 第八步：確認前端是否需改

檢查前端是否仍能：

- 顯示會員首頁
- 顯示顧問入口
- 顯示顧問業務大廳
- 顯示顧問資源中心
- 顯示推薦名單
- 顯示團隊名單

## 十、高風險提醒

正式修改前務必注意：

- 不可直接刪除 `名單狀態`
- 不可直接覆蓋舊資料
- 不可把 `已簽約` 直接當成交
- 不可把 `放棄追蹤` 全部轉成 `停止聯繫／退訂`
- 不可把 `無效名單` 直接混入 CRM 階段
- 不可在沒有備份時修改正式 Sheet
- 不可在尚未確認 Apps Script 欄位依賴前改欄位名稱
- 不可讓停止聯繫／退訂客戶繼續收到自動化行銷訊息
- 不可讓顧問誤以為已簽約未付款就是已成交

## 十一、結論與下一步

Task 03A-9 確認：

- `leads_潛在會員名單` 可以透過新增欄位方式升級為 CRM 工作表

- 舊資料轉換需保留 `原名單狀態`
- 高風險狀態需人工確認
- Apps Script 與前端需在正式修改前檢查欄位依賴
- 停止聯繫／退訂需優先保護
- 正式修改 Google Sheet 前必須先備份
